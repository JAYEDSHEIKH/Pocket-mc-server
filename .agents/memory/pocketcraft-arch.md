# PocketCraft Architecture

## Singleton process bridge
`ServerProcessManager` is a hand-rolled singleton (not Hilt-managed) that bridges `ServerForegroundService` ↔ ViewModels. Both sides hold a reference via `ServerProcessManager.instance`. Status changes flow through `allStatuses: StateFlow<Map<String, ServerStatus>>` which the service collects to persist to Room, and ViewModels observe for live UI updates.

**Why:** Hilt cannot inject into a `Service` and a `ViewModel` and share the same mutable state object without a custom scope. The singleton pattern avoids a binder/AIDL dance.

## Status persistence
`ServerForegroundService.collectStatusUpdates()` is the single authoritative path for writing process status to Room. It collects `ServerProcessManager.allStatuses` and calls `serverProfileDao.updateStatus()` on every change. No other code path should write status except on service destroy (reset stale).

## foregroundServiceType on API 34+
Android 14 (API 34, `Build.VERSION_CODES.UPSIDE_DOWN_CAKE`) requires the service type to be passed as a third argument to `startForeground()` when `foregroundServiceType` is declared in the manifest. Fixed in `ServerForegroundService.onCreate()`:
```kotlin
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.UPSIDE_DOWN_CAKE) {
    startForeground(id, notification, ServiceInfo.FOREGROUND_SERVICE_TYPE_SPECIAL_USE)
} else {
    startForeground(id, notification)
}
```

## Paper Fill API v3 (fill.papermc.io) — actual response shape
The DTOs were rewritten to match the real API (verified via live curl):

- `GET /v3/projects/paper` → `{ "project": {...}, "versions": Map<String, List<String>> }` where the map key is major version group and values are specific versions. Use `stableVersionsSorted()` helper to flatten + filter RC/pre/beta + sort numerically descending.
- `GET /v3/projects/paper/versions/{v}/builds` → **bare `List<PaperBuild>`** (no wrapper object).
- `channel` values are `"STABLE"` / `"EXPERIMENTAL"` (uppercase — comparison must be case-insensitive).
- Download URL is directly at `downloads["server:default"].url` (not constructed from build ID). The old `applicationDownloadUrl()` function is replaced by `serverJarUrl()`.
- `sha256` is nested at `checksums.sha256`, not top-level.

**Why this matters:** Using the wrong DTOs causes Moshi parse exceptions; version list never loads, server creation silently fails.

## RCON password security
Each server gets a cryptographically random 24-char alphanumeric password generated by `SecureRandom` at creation time, written to `server.properties`. Never use a hardcoded default — predictable RCON credentials are a security issue.
